# WOD Wiki Documentation

Welcome to the WOD Wiki documentation tree. This repository serves as the single source of truth for the system's architecture, design, and processes.

## Documentation Structure

- [**Plans**](./plans/) - Active and upcoming planning documents.
- [**Reports**](./reports/) - Investigation results and status reports.
- [**Audits**](./audits/) - Formal reviews of system components.
- [**Deep Dives**](./deep-dives/) - Technical explanations and implementation guides.
- [**Domain Model**](./domain-model/) - Entities, repositories, and data contracts.
- [**Design System**](./design-system/) - UI/UX patterns, layouts, and components.
- [**Testing**](./testing/) - Testing standards and guides.
- [**ADR**](./adr/) - Architectural Decision Records.
- [**Archive**](./archive/) - Historical material and completed projects.

## Contributor Guidelines

### Where does my document go?
- **Adding a new feature plan?** Put it in `docs/plans/`.
- **Reporting on a bug investigation or system state?** Put it in `docs/reports/`.
- **Explaining how a complex module works?** Put it in `docs/deep-dives/`.
- **Documenting a permanent architectural decision?** Put it in `docs/adr/`.
- **Updating UI patterns or layouts?** Put it in `docs/design-system/`.

### Rules for Documentation
1. **One Doc, One Purpose**: Avoid mixing plans with reports or deep dives.
2. **Current Truth First**: Canonical documentation should be easily discoverable.
3. **Archive Stale Material**: Move outdated documents to the `archive/` folder once they are no longer active guidance.
4. **Descriptive Naming**: Use clear, searchable filenames (e.g., `note-persistence-plan.md` instead of `persistence.md`).
