# §3.1 Classification — Unused Files as Swap Candidates, Non-Demoable Code, and Plain Dead

> Generated 2026-06-06 from `bun x knip --no-exit-code` (174 unused-file reports; 173 unique after dedup).
>
> Cross-reference: this doc is the re-classification of the 174 files listed in `docs/dead-code-analysis.md` §3.1 ("Likely Real Dead Code"). The original doc framed them as "dead code to delete"; this doc reframes them along three orthogonal axes:
>
> 1. **Interface binding** — does the file implement an interface that the active app code uses? If so, the file is a **swap candidate**: it could replace the currently-wired implementation. If not, the file stands alone.
> 2. **Demo path** — can the playground app or a storybook story render this file as a useful demonstration? (UI components yes; server-only code, type-only modules, runtime internals, test helpers — no.)
> 3. **Bucket** — the combination of the above puts each file in one of three categories below.

---

## Methodology

The classification was produced by:

1. Running `bun x knip --no-exit-code` to get the 174-file unused-file list.
2. For each file: reading the first 30 lines to find any `implements X`, `extends X`, or registry registration.
3. For interface-bearing files: `search` the repo for the interface name to determine whether the interface is referenced by active code.
4. Bucketing by combining (interface, interface-active) with (is-UI-renderable):
   - (implements active interface, not UI) → SWAP_CANDIDATE
   - (no active interface, config/tooling/test/server/runtime) → NON_DEMOABLE
   - (no active interface, renderable as UI) → PLAIN_DEAD
   - (implements inactive interface) → NON_DEMOABLE (the whole subsystem is dead)

## Update 2026-06-09: cast-stack refactor
The cast surface (`src/services/cast/`, `src/hooks/useCastSignaling.ts`, `src/components/molecules/CastButton.tsx`, `src/components/molecules/CastCallout.tsx`, `src/hooks/useChromecast.ts`, `src/hooks/__tests__/useCastSignaling.test.ts`, `src/receiver-main.tsx`) was reshaped by introducing an `ICastBackend` port (`src/services/cast/ICastBackend.ts`) and a `getCastBackend()` factory (`src/services/cast/getCastBackend.ts`). The button (`src/components/organisms/cast/CastButtonRpc.tsx`) now consumes the port. As a side effect, the following files became dead and were deleted:
| Deleted | Was classified as |
|---------|-------------------|
| `src/components/molecules/CastButton.tsx` | PLAIN_DEAD (3a) |
| `src/components/molecules/CastCallout.tsx` | PLAIN_DEAD (3a) |
| `src/hooks/useChromecast.ts` | NON_DEMOABLE (2b — hook with no consumer) |
| `src/hooks/__tests__/useCastSignaling.test.ts` | test of a now-removed hook |
| `src/receiver-main.tsx` | NON_DEMOABLE (2b — replaced by `src/receiver-rpc.tsx`) |
| `src/services/cast/CastManager.ts` | PLAIN_DEAD (3b — deprecated WebSocket relay) |
| `src/services/cast/WebRTCTransport.ts` | legacy transport, only consumed by deleted code |
| `src/services/cast/constants.ts` | PLAIN_DEAD (3b — unused cast constants) |
The barrel `src/hooks/useCastSignaling.ts` was narrowed to re-export only the cast stack's *concrete classes* (`ChromecastSenderViewSession`, `ChromecastRuntimeSubscription`, `ChromecastEventProvider`, `ClockSyncService`) and *types* (`IRpcTransport`, `RpcMessage`, `RpcWorkbenchUpdate`, etc.) consumed by downstream bridges. The `ICastBackend` port and `getCastBackend()` factory live in `src/services/cast/`.
The new shape makes the cast surface a real seam: a second adapter (`LocalTabBackend`, defined in `src/services/cast/adapters/LocalTabBackend.ts`) implements the same port and is selected at build time by `VITE_CAST_BACKEND`. The chromecast path is unchanged in behavior; the local path is exercised via the new `LocalTabBackend.test.ts` and `BroadcastChannelRpcTransport.test.ts`. An `eventRouter` (`src/services/cast/rpc/eventRouter.ts`) extracts the cast event-name → runtime-handle decision that used to be inlined in the cast button, and the cast-roundtrip integration test now uses the same router — same code path on both sides of the seam.
