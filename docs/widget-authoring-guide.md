# Widget Authoring Guide

This guide defines the Phase 1C authoring contract for editor widgets rendered from fenced markdown blocks.

## Scope

Use this guide when adding or reviewing widgets that render through:

- `src/components/Editor/extensions/widget-block-preview.tsx`
- `src/components/Editor/overlays/WidgetCompanion.tsx`
- `src/components/Editor/widgets/types.ts`

These widgets render markdown source that looks like this:

````md
```widget:my-widget
{
  "title": "Hello"
}
```
````

## Authoring Contract

### 1. Widget source is fenced markdown + JSON object body

Every widget block must use:

- an opening fence in the form ```` ```widget:<name> ````
- a closing ```` ``` ```` fence
- a JSON **object** between the fences

Rules:

- widget names use lowercase kebab-case, for example `attention-card`
- the JSON body is the canonical source of truth
- rendered React output is a view over source, not a second persistence model
- top-level arrays, strings, and numbers are not valid widget configs

### 2. Use the shared `WidgetProps` contract

Source of truth: `src/components/Editor/widgets/types.ts`

```ts
export type WidgetConfig = Record<string, unknown>

export interface WidgetProps {
  config: WidgetConfig
  rawContent: string
  sectionId: string
}
```

Field guidance:

- `config`: parsed JSON object for the widget
- `rawContent`: exact body text between the fences; use for diagnostics and debugging
- `sectionId`: stable identifier generated by `section-state.ts`; use for DOM ids, analytics hooks, or editor coordination

Keep the base contract small. Do not expand it with page-specific callbacks, router handles, or editor state unless the capability is truly generic across all widgets.

### 3. Validate widget-specific schema inside the widget module

Shared infrastructure owns only universal parsing rules:

- the block is a widget fence
- the widget name is present
- the body parses as JSON
- the parsed value is a JSON object

Everything else belongs to the widget itself.

Recommended pattern:

1. Accept `WidgetProps`
2. Narrow `config` into a widget-local interface
3. Return a visible author-facing error state when required fields are missing or malformed
4. Keep the malformed raw source visible enough to repair quickly

Reference template:

- `src/components/Editor/widgets/ExampleWidget.template.tsx`

### 4. Register widgets by name in a `WidgetRegistry`

Widgets are registered as:

```ts
const widgetComponents: WidgetRegistry = new Map([
  ['my-widget', MyWidget],
])
```

Registration guidance:

- choose a stable public name before shipping content that depends on it
- keep the exported component name descriptive and the registry key terse
- treat renames as content migrations, not cosmetic edits

### 5. Keep widgets stateless and fast

Phase 1C widgets are display-first components.

Constraints:

- no async data loading inside a widget render path
- no nested editors
- no hidden two-way sync with another persistence model
- prefer pure rendering from `config`
- keep per-widget DOM shallow and token-driven
- assume authors may place several widgets on one page; avoid timers, observers, or layout thrash unless essential

Performance budget guidance:

- typical playground notes should stay around `0–3` widgets
- dense demo templates may go higher, but any page heading toward `8+` widgets needs a smoke check for scroll and mount cost

### 6. Accessibility is part of the contract

Every widget must:

- use semantic HTML where possible
- preserve color contrast in light and dark themes
- expose accessible names for interactive controls
- avoid hover-only meaning
- degrade clearly when config is invalid

### 7. Edit behavior and interaction model

Widgets rendered through `widget-block-preview.tsx` support an inline edit lifecycle. Authors should understand this model when designing widgets that may need config tuning inside the editor.

#### Edit icon

- A pencil icon appears on **hover** or **keyboard focus** of the widget block
- Clicking the icon enters edit mode (or saves, or undoes, depending on state)
- The icon is hidden by default and transitions in over `200ms`

#### Edit mode

- Entering edit mode replaces the widget preview with a monospace textarea containing the raw JSON body
- The textarea auto-focuses and places the cursor at the end of the content
- The border switches to an emerald tint to indicate active editing

#### Auto-save on blur

- Valid JSON changes are saved automatically when focus leaves the widget block
- Saved content is normalized back into the fenced body with a trailing newline so section parsing remains stable
- The widget preview re-renders immediately after save

#### Error inlay and recovery

- Invalid JSON does **not** persist on blur
- An error inlay appears below the textarea showing the parse error
- The action button switches to an undo icon
- Clicking undo discards unsaved edits and restores the last valid preview
- Typing valid JSON while the error inlay is visible clears the error immediately

#### Keyboard shortcuts

| Shortcut | Context | Action |
| --- | --- | --- |
| `ArrowUp` / `ArrowDown` | Editor cursor near widget | Treats widget as a single line; moves to previous/next paragraph without entering edit mode |
| `Enter` | Widget has keyboard focus | Enters edit mode and focuses the textarea |
| `Escape` | Inside edit-mode textarea | Discards unsaved changes and exits edit mode |
| `Ctrl/Cmd + Enter` | Inside edit-mode textarea | Saves valid JSON and exits edit mode |

#### Focus management

- Widget focus is tracked independently of editor selection
- Moving the editor cursor into the fenced source region reveals raw markdown (existing CM6 behavior)
- Moving the editor cursor past the widget block via arrow keys does not trigger edit mode
- Tab navigation moves into the widget block, through the action button, and out to the next focusable element

### 8. Document the widget where authors will look

Before shipping a new widget:

- add or update an entry in `docs/widget-catalog.md`
- keep JSDoc on the widget component accurate
- add a usage snippet that authors can paste into markdown
- link the widget to design or ADR context when the schema is non-obvious

## Recommended Build Pattern

1. Copy `src/components/Editor/widgets/ExampleWidget.template.tsx`
2. Rename the component and config interface
3. Define a small parser that narrows `WidgetConfig`
4. Render a clear invalid-config state
5. Register the widget in the page-level `WidgetRegistry`
6. Add a catalog entry and example snippet
7. Verify:
   - widget renders when unfocused
   - raw source appears when the cursor enters the block
   - malformed config produces an explicit error state
   - edit icon appears on hover and focus
   - entering edit mode shows the raw JSON textarea
   - valid changes auto-save on blur
   - invalid JSON shows error inlay with undo recovery
   - keyboard navigation (ArrowUp/ArrowDown, Enter, Escape) works as documented
