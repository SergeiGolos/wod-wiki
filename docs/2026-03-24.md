# Canvas Page — Markdown DSL

> Canvas Pages are authored as ordinary markdown files. Special fenced code blocks and heading attributes give the renderer enough information to mount embedded views, wire commands, and control scroll behaviour — without writing any React. **This DSL applies only to Canvas Pages** (`template: canvas`). Other templates (List + Detail Panel, Temporal List, Note Workspace) are driven by routing and data, not by inline markdown syntax.

---

## Blocks

### `view` — Declare a named interactive panel

```view
name:    <identifier>          # unique name; used as target by command/button blocks
state:   browse | note | track | review   # initial display mode
source:  <note-id> | query:<expr>         # data source — a note ID or an index-db query
runtime: in-memory | index-db             # in-memory = ephemeral; index-db = persistent
launch:  host | dialog | window           # host = inline in page; dialog = modal; window = popup
align:   left | right | center | full     # layout hint within the section
width:   <css-value>                      # optional width override, e.g. 50% or 400px
```

| Property  | Required | Values                           | Notes                                                                     |
| --------- | -------- | -------------------------------- | ------------------------------------------------------------------------- |
| `name`    | yes      | identifier string                | Must be unique on the page; used as the `target` in command/button blocks |
| `state`   | no       | `browse` `note` `track` `review` | Default: `browse`                                                         |
| `source`  | no       | note ID or `query:<expr>`        | Omit to start empty; `query:today` queries IndexDB for today's entries    |
| `runtime` | no       | `in-memory` `index-db`           | Default: `in-memory`; use `index-db` to read/write persisted data         |
| `launch`  | no       | `host` `dialog` `window`         | Default: `host`                                                           |
| `align`   | no       | `left` `right` `center` `full`   | Default: `full`                                                           |

---

### `command` — Send commands when a heading enters the viewport

Placed anywhere inside a heading section. Fires the pipeline as soon as the section scrolls into view.

```command
target:   <view-name>    # name of the view to address
pipeline:
  - <action>: <value>    # ordered list of pipeline steps
```

---

### `button` — Send commands on click

```button
label:    <display text>
target:   <view-name>
pipeline:
  - <action>: <value>
```

---

### Shared pipeline actions

Both `command` and `button` drive the same command bus.

| Action | Value | Effect |
|--------|-------|--------|
| `set-state` | `browse` \| `note` \| `track` \| `review` | Switches the view's display mode |
| `set-source` | note ID or `query:<expr>` | Loads new data into the view |
| `navigate` | route path | Pushes a new browser route |
| `launch` | `host` \| `dialog` \| `window` | Changes where the view renders |
| `set-filter` | filter expression | Applies a filter to lists inside the view |
| `focus` | section-id | Scrolls the view to a specific section |

---

## Heading attributes

Add attributes directly to a heading (`# Title {attr}`) to control scroll behaviour.

| Attribute | Effect |
|-----------|--------|
| `{sticky}` | Pins the heading to the top of the viewport while its section is visible |
| `{sticky content}` | Pins both the heading *and* all content in the section (full section stays in view) |
| `{full-bleed}` | Stretches the section background edge-to-edge past the content column |
| `{dark}` | Forces dark theme for this section regardless of user preference |

---

### 2. List + Detail Panel

|                 |                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| --------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Name**        | List + Detail Panel                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| **Code basis**  | `DocsPageShell` / `CollectionsPage` layout — `src/panels/page-shells/DocsPageShell.tsx`, `src/app/pages/CollectionsPage.tsx`                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| **Description** | Two-zone layout: a sticky **Detail Panel** at the top that contextualises the current grouping (a doc category summary, a collection header, etc.) and a scrollable **List** below where individual items are browsed and selected. Clicking an item may expand an inline preview or navigate to a detail route. The Detail Panel organism is swappable — that is the only structural difference between browsing docs, browsing all collections, and browsing one collection's items. Replaces the former Card List Page and Collection List Page as a single unified template. |

**Routes using this template**

| Route                | Page name         | Detail Panel organism                                  |
| -------------------- | ----------------- | ------------------------------------------------------ |
| `/syntax`            | Syntax index      | Doc category summary (metadata + description)          |
| `/collections`       | Collections index | Summary stats / filter header for all collections      |
| `/collections/:slug` | Collection detail | `CollectionCard` with description + membership actions |

**Organisms on this template**

| Organism | Role |
|---------|------|
| `StickyNavPanel` | Persistent top navigation bar |
| `CollectionBrowsePanel` *(swappable Detail Panel slot)* | Contextual header describing the current grouping |
| `ListOfNotes` / `CollectionItemList` | Scrollable item list in the main body |
| `ListFilter` | Filter + search bar above the list |
| `NotePreview` *(inline detail)* | Expands below a selected list item to show content preview |
| `AnalyzePanel` | Optional secondary panel for comparison / analytics |

**Markdown example**

````markdown
---
template: list-detail
---

# Collections {sticky}

```view
name:    detail
state:   browse
source:  query:all-collections
runtime: index-db
launch:  host
align:   full
```

```command
target: detail
pipeline:
  - set-source: query:all-collections
  - set-state: browse
```

## Items

```view
name:    list
state:   browse
source:  query:collection-items
runtime: index-db
launch:  host
align:   full
```

```button
label:  New Entry
target: list
pipeline:
  - set-state: note
  - launch: dialog
```
````

> For `/collections/:slug` the frontmatter `source` resolves to the specific collection;
> the Detail Panel view loads that collection's metadata. The list view filters to its items.
> For `/syntax` the source resolves to the docs index query; for `/syntax/:topic` it resolves
> to the specific doc ID and the detail view renders in `note` state (read-only).

### 3. Temporal List

| | |
|--|--|
| **Name** | Temporal List |
| **Code basis** | `CalendarPageShell` / `JournalPageShell` — `src/panels/page-shells/CalendarPageShell.tsx`, `src/panels/page-shells/JournalPageShell.tsx` |
| **Description** | A time-period header pinned at the top that provides the grouping context (month calendar or week strip), with a type-filtered list of matching entries (notes, wod blocks, results) scrolling below. Month and Week share identical structural contracts — only the **Period Header** organism swaps. This allows a single template to handle both granularities without maintaining two separate shells. |

**Routes using this template**

| Route | Page name | Period Header organism |
|-------|-----------|----------------------|
| `/month/:yyyy-mm` | Month | `CalendarWidget` — full month grid with day dots |
| `/week/:yyyy-mm-dd` | Week | Week strip — 7-day row with activity indicators |

**Organisms on this template**

| Organism | Role |
|---------|------|
| `StickyNavPanel` | Persistent top navigation bar |
| `CalendarWidget` *(month)* or `WeekStripHeader` *(week)* | **Period Header** — swappable organism that sets the time scope |
| `CollectionsFilter` | Toggle bar: filter list by note / wod block / result |
| `HistoryPostList` | Scrollable list of entries matching the selected period and filter |
| `NewPostButton` | FAB / action button to create a new entry in this period |

**Markdown example**

````markdown
---
template: temporal-list
period:   month          # month | week — selects which Period Header organism mounts
---

# {{period-label}} {sticky content}

```view
name:    period-header
state:   browse
source:  query:period-entries
runtime: index-db
launch:  host
align:   full
```

```command
target: period-header
pipeline:
  - set-source: query:period-entries
  - set-filter: type=all
```

## Entries

```view
name:    entry-list
state:   browse
source:  query:period-entries
runtime: index-db
launch:  host
align:   full
```

```button
label:  New Entry
target: entry-list
pipeline:
  - set-state: note
  - launch: dialog
```
````

> The `period` frontmatter key tells the renderer which Period Header organism to mount
> (`CalendarWidget` for month, `WeekStripHeader` for week). Route params (`yyyy-mm` /
> `yyyy-mm-dd`) are injected into the `query:period-entries` source at runtime. Switching
> the filter buttons fires `set-filter` commands back at `entry-list`.

## Deep Linked State Management (`nuqs`)

To ensure the application state is shareable and persists through refreshes, we utilize `nuqs` for type-safe search parameter management. This moves "transient" UI state into the URL, enabling deep-linking into specific UI configurations.

### Global State Mappings

| State           | Param   | Type                          | Description                                                                                      |
| --------------- | ------- | ----------------------------- | ------------------------------------------------------------------------------------------------ |
| Palette Query   | `q`     | `string`                      | The active search string in the command palette.  If present the command pallete should be open. |
| Active Category | `cat`   | `string`                      | The selected collection category for filtering.                                                  |
| Sidebar Focus   | `sb`    | `string`                      | Which accordion is expanded (`collections`, `results`, `recent`).                                |
| Visual Theme    | `theme` | `light` \| `dark` \| `system` | Overrides the stored theme preference.                                                           |
| Debug Mode      | `debug` | `boolean`                     | Enables verbose logging and inspector overlays.                                                  |

### Implementation Strategy

1. **Adapter Wiring**: The `NuqsAdapter` is wrapped around the `Routes` in `App.tsx`.
2. **Hook Migration**: Replace `useState` with `useQueryState` for the identified parameters.
3. **Derived Strategies**: Instead of storing complex `Strategy` objects in state, store a `strategyType` string in the URL and use a `useMemo` hook to instantiate the strategy object based on that type and other URL params (like `cat` or `q`).

### Example: Search Strategy Link

A link to the Kettlebell collection in the search palette:
`/#/workout/General/Home?cmd=true&cat=Kettlebell&strategy=collection`

### Benefits
- **Reproducibility**: Bug reports can include a URL that reproduces the exact UI state.
- **Navigation**: The browser back button naturally closes the command palette or sidebar if they are tracked in the URL.
- **Sharing**: Users can share "pre-filtered" views of the wiki or specific playground configurations.
