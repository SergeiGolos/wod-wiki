# Canvas Page — Markdown DSL

> Canvas Pages are authored as ordinary markdown files. Special fenced code blocks and heading attributes give the renderer enough information to mount embedded views, wire commands, and control scroll behaviour — without writing any React. **This DSL applies only to Canvas Pages** (`template: canvas`). Other templates (List + Detail Panel, Temporal List, Note Workspace) are driven by routing and data, not by inline markdown syntax.

---

## Blocks

### `view` — Declare a named interactive panel

```view
name:    <identifier>          # unique name; used as target by command/button blocks
state:   browse | note | track | review   # initial display mode
source:  <note-id> | query:<expr>         # data source — a note ID or an index-db query
runtime: in-memory | index-db             # in-memory = ephemeral; index-db = persistent
launch:  host | dialog | window           # host = inline in page; dialog = modal; window = popup
align:   left | right | center | full     # layout hint within the section
width:   <css-value>                      # optional width override, e.g. 50% or 400px
```

| Property  | Required | Values                           | Notes                                                                     |
| --------- | -------- | -------------------------------- | ------------------------------------------------------------------------- |
| `name`    | yes      | identifier string                | Must be unique on the page; used as the `target` in command/button blocks |
| `state`   | no       | `browse` `note` `track` `review` | Default: `browse`                                                         |
| `source`  | no       | note ID or `query:<expr>`        | Omit to start empty; `query:today` queries IndexDB for today's entries    |
| `runtime` | no       | `in-memory` `index-db`           | Default: `in-memory`; use `index-db` to read/write persisted data         |
| `launch`  | no       | `host` `dialog` `window`         | Default: `host`                                                           |
| `align`   | no       | `left` `right` `center` `full`   | Default: `full`                                                           |

---

### `command` — Send commands when a heading enters the viewport

Placed anywhere inside a heading section. Fires the pipeline as soon as the section scrolls into view.

```command
target:   <view-name>    # name of the view to address
pipeline:
  - <action>: <value>    # ordered list of pipeline steps
```

---

### `button` — Send commands on click

```button
label:    <display text>
target:   <view-name>
pipeline:
  - <action>: <value>
```

---

### Shared pipeline actions

Both `command` and `button` drive the same command bus.

| Action | Value | Effect |
|--------|-------|--------|
| `set-state` | `browse` \| `note` \| `track` \| `review` | Switches the view's display mode |
| `set-source` | note ID or `query:<expr>` | Loads new data into the view |
| `navigate` | route path | Pushes a new browser route |
| `launch` | `host` \| `dialog` \| `window` | Changes where the view renders |
| `set-filter` | filter expression | Applies a filter to lists inside the view |
| `focus` | section-id | Scrolls the view to a specific section |

---

## Heading attributes

Add attributes directly to a heading (`# Title {attr}`) to control scroll behaviour.

| Attribute | Effect |
|-----------|--------|
| `{sticky}` | Pins the heading to the top of the viewport while its section is visible |
| `{sticky content}` | Pins both the heading *and* all content in the section (full section stays in view) |
| `{full-bleed}` | Stretches the section background edge-to-edge past the content column |
| `{dark}` | Forces dark theme for this section regardless of user preference |

---

> Each `##` heading becomes a `ScrollSection`; `{sticky}` pins it during scroll;
> `view`, `command`, and `button` blocks wire the embedded interactive panels.


# Pages

## How routing works

All routes funnel through a single `AppContent` component. On every render, `findCanvasPage(pathname)` performs an O(1) Map lookup against every canvas markdown file loaded at build time. If a match is found, `CanvasPage` renders. If not, `AppContent` falls through to the appropriate hardcoded component.

**Canvas pages are sourced from two directory trees, both imported at build time via `import.meta.glob`:**

| Directory | Glob | Route derivation |
| --------- | ---- | ---------------- |
| `markdown/canvas/` | `markdown/canvas/**/*.md` | `route:` frontmatter key (required) |
| `markdown/collections/*/README.md` | `markdown/collections/**/README.md` | Auto-derived from folder name → `/collections/{slug}` |

### 1. Canvas Page

| | |
|--|--|
| **Name** | Canvas Page |
| **Code** | `playground/src/canvas/CanvasPage.tsx` + `parseCanvasMarkdown.ts` |
| **Source files** | `markdown/canvas/**/*.md` and `markdown/collections/**/README.md` |
| **Description** | Scroll-driven editorial layout. Sections are parsed from markdown headings. Each section can have a sticky editor panel (a `view` block) on the left or right, prose in the main column, embedded buttons, and scroll-triggered command pipelines. When a section has **no `view` block**, the full viewport width is used with rich markdown prose rendering (tables, images, code blocks, frontmatter cards, file links). |

**Two layout modes within Canvas:**

| Mode | When | Column split |
|------|------|-------------|
| Two-column | Section contains a `view` block | 40% prose + 60% sticky editor (`NoteEditor` or `RuntimeTimerPanel`) |
| Full-width | Section has no `view` block | 100% width, `max-w-4xl` content container with rich `CanvasProse` markdown |

**Heading attributes:**

| Attribute | Effect |
|-----------|--------|
| `{sticky}` | Fires the section's `command` pipeline when it enters the viewport |
| `{dark}` | Dark tint background for this section |
| `{full-bleed}` | Section stretches edge-to-edge; content centred at `max-w-md` |

**DSL blocks (stripped from prose, never rendered as code):**

| Block | Purpose |
|-------|---------|
| `` ```view `` | Declares the sticky editor panel — source file, alignment, inline buttons |
| `` ```command `` | Pipeline that fires automatically when the section scrolls into view |
| `` ```button `` | Clickable button that fires a pipeline on click — rendered below section prose |

**Pipeline actions:**

| Action | Effect |
|--------|--------|
| `set-source: <path>` | Swaps the editor content (supports `markdown/canvas/`, `markdown/collections/`, `wod/` paths) |
| `set-state: track` | Runs the compiled WOD block (opens in `view`, `dialog`, or `route` mode) |
| `navigate: <route>` | Pushes a new browser route |

**Collection README pattern:**

Collection pages (`markdown/collections/{slug}/README.md`) use Canvas as a structured tutorial/walkthrough format:
- No `route:` frontmatter — route is auto-derived from folder name at build time
- Prose sections introduce each workout in the collection
- Each workout gets its own `##` heading with description, the WodScript as a fenced `wod` block (preserved as a code block in the prose via `CanvasProse`), and optionally a `button` to load/run it
- The sticky editor panel (`view` block) loads workout content on scroll via `command` blocks

**Markdown authoring example:**

````markdown
---
template: canvas
route: /syntax/basics
---

# The Basics {sticky}

```view
name:    demo
state:   note
source:  markdown/canvas/syntax/basics.md
runtime: in-memory
align:   right
button:  Run It | set-state: track | open: dialog
```
````

## Workbench / Note Context

The **Workbench** is the page-level context that holds the currently loaded note, any compiled runtimes derived from it, and the results produced when those runtimes run. It is not a single component — it is a pattern expressed differently depending on the route's persistence requirements.

### `inMemory` workbench (Canvas pages)

Used on all canvas-routed pages (collections, syntax, home, getting-started, etc.). Workout state lives inside React component state in `CanvasPage`. When a button pipeline fires `open: route`, the compiled `WodBlock` is handed off via `pendingRuntimes` — a plain module-scoped `Map<string, { block: WodBlock; noteId: string }>` in `playground/src/runtimeStore.ts`. The key is a UUID; `TrackerPage` reads and immediately deletes it. Nothing is persisted between reloads.

### `indexDB` workbench (Playground + Journal pages)

Used on routes where the note must survive page refreshes. Notes are loaded and saved through `usePlaygroundContent` (`playground/src/hooks/usePlaygroundContent.ts`), which follows an IndexedDB-first strategy:

1. On mount: load from IndexedDB by `(category, name)` key — fall back to the bundled MD template and seed IndexedDB.
2. On edit (`onChange`): persist to IndexedDB immediately so block IDs remain stable across sessions.
3. Results from completed workouts are saved via `indexedDBService.saveResult()` and retrieved by `ReviewPage` via `indexedDBService.getResultById(runtimeId)`.

| Route | Category | Name (slug) | Notes |
|-------|----------|-------------|-------|
| `/playground/:id` | `playground` | UUID generated once per instance | Created fresh via New Playground action |
| `/journal/:id` | `journal` | `yyyy-mm-dd` (daily) | Multiple entries on the same day use a `yyyy-mm-dd/HH:MM` suffix |

## Deep Linked State Management (`nuqs`)

To ensure the application state is shareable and persists through refreshes, we utilize `nuqs` for type-safe search parameter management. This moves "transient" UI state into the URL, enabling deep-linking into specific UI configurations.

### Global State Mappings

| State           | Param   | Type                          | Description                                                                                      |
| --------------- | ------- | ----------------------------- | ------------------------------------------------------------------------------------------------ |
| Palette Query   | `q`     | `string`                      | The active search string in the command palette.  If present the command pallete should be open. |
| Active Category | `cat`   | `string`                      | The selected collection category for filtering.                                                  |
| Sidebar Focus   | `sb`    | `string`                      | Which accordion is expanded (`collections`, `results`, `recent`).                                |
| Visual Theme    | `theme` | `light` \| `dark` \| `system` | Overrides the stored theme preference.                                                           |
| Debug Mode      | `debug` | `boolean`                     | Enables verbose logging and inspector overlays.                                                  |

### Implementation Strategy

1. **Adapter Wiring**: The `NuqsAdapter` is wrapped around the `Routes` in `App.tsx`.
2. **Hook Migration**: Replace `useState` with `useQueryState` for the identified parameters.
3. **Derived Strategies**: Instead of storing complex `Strategy` objects in state, store a `strategyType` string in the URL and use a `useMemo` hook to instantiate the strategy object based on that type and other URL params (like `cat` or `q`).

### Example: Search Strategy Link

A link to the Kettlebell collection in the search palette:
`/#/workout/General/Home?cmd=true&cat=Kettlebell&strategy=collection`

### Benefits
- **Reproducibility**: Bug reports can include a URL that reproduces the exact UI state.
- **Navigation**: The browser back button naturally closes the command palette or sidebar if they are tracked in the URL.
- **Sharing**: Users can share "pre-filtered" views of the wiki or specific playground configurations.
