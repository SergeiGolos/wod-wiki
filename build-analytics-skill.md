# Building Inline Analytics & Trackers

This skill provides guidelines for building features around the Inline Analytics Engine and Tracker components. It outlines the core data structures available and how to hook into the runtime lifecycle.

## Core Architecture

The runtime produces an output stream of `IOutputStatement` objects. 
1. **Analytics Processes** intercept these statements, aggregate data, and inject new derived metrics back into the statement.
2. **Trackers & UI Components** consume the statements and blindly render the metrics using components like `MetricVisualizer`. 

Because analytics are pre-calculated and injected as standard metrics, the trackers don't need to know the difference between parsed parameters and derived analytics.

---

## The Data Model

### 1. `IOutputStatement`
This is the fundamental unit of runtime execution and the primary object you'll interact with.

```typescript
interface IOutputStatement {
  outputType: 'segment' | 'milestone' | 'system' | 'event' | 'analytics';
  timeSpan: TimeSpan;                 // Start and end timestamps
  metrics: IMetric[];                 // Array of all collected/derived metrics
  
  // Helpers
  getMetric(type: MetricType): IMetric | undefined;
  getDisplayMetrics(filter?: MetricFilter): IMetric[];
  hasMetric(type: MetricType): boolean;
  
  // Deprecated (Do not use for new analytics; pull from `.metrics` instead)
  elapsed: number;
  total: number;
}
```

### 2. `IMetric`
The standardized shape for all data points—whether they come from the parser, the runtime clock, or your analytics process.

```typescript
interface IMetric {
  type: MetricType;        // What this measures (e.g., Rep, Distance, Elapsed)
  value?: unknown;         // The numeric or string value
  image?: string;          // Human-readable label/display string
  origin: MetricOrigin;    // Where it came from (parser, runtime, analyzed, etc.)
  unit?: string;           // Optional unit (e.g., 'kg', 'm')
  timestamp?: Date;        // When it was recorded
}
```

### 3. Key `MetricType`s
When writing analytics, you will typically filter by these types:
- **`Duration`**: Parser-defined planned target (e.g., a 5-minute AMRAP).
- **`Spans` / `Elapsed` / `Total`**: Execution time data. `Elapsed` is the active time (excluding pauses), `Total` is wall-clock time.
- **`Rep`**: Repetitions completed or targeted.
- **`Distance` / `Resistance`**: Movement metrics (values are often objects like `{ amount: 100, units: 'm' }`).
- **`Effort`**: Pacing labels (e.g., "hard", "easy").
- **`Metric` / `Label`**: Generic types often used by the Analytics Engine to inject summary text (e.g., "Total Volume: 5000kg").

### 4. `MetricOrigin`
Helps resolve precedence and identifies the source of the metric:
- `'parser'`: Written in the script (e.g., "10 reps")
- `'runtime'`: Generated by block execution (e.g., elapsed timer)
- `'analyzed'`: Injected by an `IAnalyticsProcess`.
- `'user'`: Explicit user override or logging.

## Writing an Analytics Process

An analytics process must implement `IAnalyticsProcess`. It processes statements as they are emitted, maintains internal state (like a running sum), and yields final summary statements at the end of the workout.

```typescript
import { IAnalyticsProcess } from '../contracts/IAnalyticsEngine';
import { IOutputStatement, OutputStatement } from '../models/OutputStatement';

export class CustomAnalyticsProcess implements IAnalyticsProcess {
  public readonly id = 'my-custom-analytics';
  private total = 0;

  process(output: IOutputStatement): IOutputStatement {
    // 1. Filter for completed segments
    if (output.outputType === 'segment') {
      
      // 2. Extract metrics to analyze
      const repMetric = output.getMetric(MetricType.Rep);
      if (typeof repMetric?.value === 'number') {
        this.total += repMetric.value;
      }

      // 3. Inject new derived/analyzed metrics inline
      output.metrics.push({
        type: MetricType.Metric,
        image: `Running Total: ${this.total}`,
        value: this.total,
        origin: 'analyzed',
        timestamp: new Date()
      });
    }
    return output;
  }

  finalize(): IOutputStatement[] {
    // 4. Return summary statement(s) when the session ends
    return [new OutputStatement({
      outputType: 'analytics',
      timeSpan: new TimeSpan(Date.now(), Date.now()),
      sourceBlockKey: 'analytics-engine',
      stackLevel: 0,
      metrics: [
        { type: MetricType.Label, value: `Final Total: ${this.total}`, origin: 'analyzed' }
      ]
    })];
  }
}
```

## Trackers & UI Views

When building UI components to display runtime execution or history:

1. **Active Tracking (Runtime)**:
   Subscribe to a block's metric memory tiers (`display`, `promote`, `private`). 
   *Example: Use `block.getMetricMemoryByVisibility('display')`.*

2. **History / Post-Workout Views**:
   You will receive an array of completed `IOutputStatement[]`. Iterate through the segments, and use `statement.getDisplayMetrics()` to feed standard components like `<MetricVisualizer />` or `<MetricSourceRow />`.

   Because the Analytics Engine already pushed "Running Total" or "Velocity" into the `metrics` array with the origin `'analyzed'`, you do not need to re-calculate these in the React component. Just render the metrics list.
